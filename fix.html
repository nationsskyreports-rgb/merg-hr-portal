المتغيرات الحساسة والروابط

أين توجد وكيف تغيّرها

رابط Supabase و API Key

موجودين في أول سطر في ملف app.js:

const sb = window.supabase.createClient( 'https://wnuzxosjjhucxnlaptbf.supabase.co', // ← رابط المشروع (Project URL) 'eyJhbGciOiJIUzI1NiIs...' // ← Anon Key (مش Secret Key!) );
تحذير: الـ Anon Key ده مش سري — هو مصمم يظهر في الكود. الـ Service Role Key هو السري ومش يستخدم هنا أبداً. لو عايز تغيّر المشروع، غيّر الرابطين دول بس.

isAdmin = emp.email === 'admin@merge.com'; // ← غيّر ده لبريد الإدمن بتاعك

لو عايز تغيّر بريد الإدمن، غير القيمة بين علامات التنصيص.

STORAGE_BUCKET: 'employee-files' // ← نفس اسم الـ Bucket في Supabase Storage

إنشاء جدول جديد في Supabase

خطوة بخطوة معRLS

إنشاء الجدول

1

افتح Supabase → Table Editor → New Table

2

أضف الأعمدة

كل جدول لازم يكون فيه id (uuid, default: gen_random_uuid()) و created_at (timestamp, default: now())

3

أضف Foreign Key

لو الجدول مرتبط بموظف، أضف عمود employee_id (uuid) واعمله Foreign Key لجدول employees

4

فعّل RLS

لازم تفعّل Row Level Security — بدونها أي حد يقدر يقرأ كل البيانات!

إعداد RLS (مهم جداً)

1

افتح Authentication → Policies

من جانب الجدول → RLS → Enable RLS (فعّله)

2

أنشئ Policy للقراءة

New Policy → Using custom template → اختر "For authenticated users only" → غير الـ WHERE حسب الحاجة

3

أنشئ Policy للكتابة/التعديل

نفس الطريقة — اختار الـ operations المطلوبة

// مثال Policy لقراءة ملاحظات الأداء (الموظف يشوف بتاعته بس) CREATE POLICY "Users can read own notes" ON performance_notes FOR SELECT USING (auth.uid() = employee_id); // مثال Policy للإدمن يقرأ الكل CREATE POLICY "Admin can read all notes" ON performance_notes FOR SELECT USING (true);
خطأ شائع: ناس كتير بتنسى تفعّل RLS! بدون RLS أي حد يقدر يقرأ ويعدّل كل البيانات. تأكد دايماً إن RLS مفعّل لكل جدول فيه بيانات حساسة.

الأمان

نقاط مهمة تحمي النظام

النظام بيعملها صح (محتاج تتأكد بس)

  • Anon Key مش سري: مصمم يظهر في الكود — مش مشكلة
  • Service Role Key: مش مستخدم في الكود أبداً — تأكد من كده
  • كلمات المرور مشفرة: Supabase بيخزّنها بـ bcrypt — حتى مدير Supabase مش يشوفها
  • روابط الملفات مؤقتة: 60 ثانية بس — مش دائمة

لازم تتأكد منها

  • RLS مفعّل لكل جدول — بدون كده أي حد يقدر يقرأ كل البيانات
  • مش بتستخدم service_role في الـ Frontend — لو لاقيه، احذفه فوراً
  • Email confirmation مفعّل — في Supabase → Auth → Settings
  • مش بيحفظ بيانات حساسة في Local Storage — غير lang و dark فقط
قاعدة ذهبية: لو مش فاهم حاجة في الأمان، متلمسهاش. الأمان الحالي كويس ومبني صح. التعديلات الغلط ممكن تفتح ثغرات.

قائمة فحص دورية

اعمل دي كل شهر أو بعد كل تعديل كبير

فحص البيانات

  • تأكد إن RLS مفعّل لكل الجداول

  • راجع أرصدة الإجازات — متطابقة مع عدد الإجازات الموافق عليها
  • تأكد إن إحداثيات المكتب والنطاق صحيحين

فحص الأداء

  • افتح الصفحة في Chrome واضغط F12 → Lighthouse → Run — اللون الأخضر أكتر من 80
  • جرب تسجيل حضور من موبايل حقيقي — يشتغل بدون مشاكل
  • جرب تصدير CSV — الملف يفتح صح في Excel
  • جرب رفع ملف كبير (9MB) — يرفع بدون خطأ
  • اختبر مع أكثر من مستخدم في نفس الوقت

فحص الأمان

  • افتح Console — مفيش أخطاء حمراء
  • افتح Network — مفيش طلبات فاشلة (4xx/5xx)
  • جرّب تسجيل دخول ببيانات غلط — يرفض صح
  • جرّب دخول موظف A يشوف بيانات موظف B — لازم يفشل
  • تأكد إن Service Role Key مش موجود في أي ملف JS

فحص النسخ الاحتياطي

  • نسخة احتياطية أخيرة منشأة خلال آخر 7 أيام
  • النسخة محفوظة في مكان آمن (مش على نفس الجهاز)
  • جرّب استعادة من نسخة قديمة — بتشتغل

أسئلة شائعة للصيانة

محتاج سيرفرBackend؟

لا. النظام كله Frontend + Supabase. Supabase هو الـ Backend. محتاج سيرفر استضافة فقط للملفات الثابتة (HTML/CSS/JS) — وده ممكن مجاني على Netlify أو Vercel.

عايز أنقل المشروع لـ Supabase جديد — إيه الخطوات؟
  • أنشئ مشروع جديد في Supabase
  • أنشئ كل الجداول (من الأفضل تستخدم SQL dump)
  • أنشئ Storage Bucket اسمه employee-files
  • فعّل RLS وضبط Policies
  • غيّر الرابط والـ Anon Key في app.js
  • أنشئ حساب admin@merge.com في Auth واضيفه في جدول employees
عايز أضيف لغة تالتة (فرنسي مثلاً)؟

في كائن T في app.js، أضف قسم ثالث:

fr: { welcome_back: 'Bienvenue', sign_in: 'Connexion', ... },

وعدّل دالة toggleLang() لتدعم التبديل بين 3 لغات.

النظام بيشتغل أوفلاين؟

جزئياً. الـ Service Worker (sw.js) بيخزّن الملفات محلياً بعد أول زيارة. لكن العمليات (حضور، إجازات...) محتاج إنترنت عشان توصل لـ Supabase. لو الإنترنت قطع، المستخدم يقدر يفتح التطبيق لكن مش يقدر يسجّل حضور أو يبعت طلب.

عايز أغيّر اسم النظام من MERGE HR لاسم الشركة؟

ابحث عن MERGE في 3 أماكن: index.html (في الـ HTML و manifest.json)، app.js (في renderLogin و emp-header)، admin.js (في hrApp header). استبدلها بالاسم الجديد.

المشروع ده ممكن يتسع كام موظف؟

عملياً Supabase المجاني بيستحمل ملايين الصفوف. النظام مش هيواجه مشكلة حجم حتى لآلاف الموظفين. اللي ممكن يحصل هو بطء في عرض البيانات لو بتجيب آلاف سجلات مرة واحدة — والحل إنك تستخدم .limit() زي ما معمول فعلاً (60 سجل للسجل، 50 للإجازات...).
